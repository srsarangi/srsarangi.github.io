Architectural Support For Handling Jitter in Parallel Real Time App-lications

Smruti Ranjan Sarangi
 srsarangi@cse.iitd.ac.in

Hello all, welcome to our project home-page on Jitter-Management. We are primarily interested in characterizing, identifying and mitigating jitter in parallel real-time applications for large general purpose multi-core platforms. "Jitter" is defined as the non-determinism in program execution introduced by external agents like the operating system or competing hardware threads. Currently, our team has following team members.

Issues with Parallel Real-Time Applications on General Purpose Processors

Parallel soft real time applications like network packet processing, multimedia applications and stream processing have been traditionally run on custom ASIC’s and FPGA’s. Owing to Moore’s Law, today’s general purpose multicore processors also enjoy some of their advantages like high throughput and low power consumption. Consequently, there is a growing trend of porting applications hitherto run on customized hardware to general purpose multicore platforms.
Some of the prior works have established that even with the additional computational power, it is hard to guarantee soft real-time guarantees for real-time applications. Major sources of interference identified are :
  • OS activities like kernel threads, daemons, handling timer-interrupts
  • Simultaneous Threads competing for shared resources like memory bus, cache lines, etc.
  • Temperature Management Schemes like Dynamic Voltage Frequency Scaling (DVFS) and Processor Throttling
  • Overheads in providing virtual machine abstractions
All of these mechanisms introduce non-determinism into program execution, which is collectively called Jitter. Prior work on jitter suggests that even small values of jitter can slowdown large parallel programs by upto 400%.

Managing Jitter

Our work is primarily focused on characterizing various sources of jitter, evaluating their impact on performance, and mitigating jitter by augmenting existing hardware and software. Modern parallel applications make use of lot of thread synchronization primitives, signal, etc. Such synchronization calls typically make use of interrupts/system calls. Our initial studies on synchronization primitives suggest that they are the biggest contributors of jitter for modern parallel applications. We plan to modify the kernel of a commodity OS like Linux or the glibc libraries to log the beginning and end of such calls. Since, we are considering parallel real-time applications, we also expect higher contention for shared resources such as the bus and caches, which would result in higher jitter. Using a full system simulator, we plan to characterize all of these sources of jitter.
To mitigate the jitter introduced by synchronization operations, we have proposed the use of a novel passive, and non-intrusive hardware unit, which monitors synchronization calls. A finite state machine processes the events collected by this dedicated hardware unit, and subsequently it yields an estimate of the synchronization jitter.

Figure 1: Jitter Unit

We also wish to propose several other hardware counters to measure the effects of other sources of jitter such as shared buses, and caches. On the basis of these measurements, we wish to look at voltage-frequency scaling schemes to compensate for the jitter. Along with voltage-frequency schemes, we wish to propose a novel memory hierarchy for just running operating system code. This hierarchy will have its own set of caches and virtual memory management modules. These will be designed to minimize destructive interference in the last level cache or main memory.

Current Status And Planned Experiments

We plan to collect memory footprints of a standard commodity OS like Linux along with some standard parallel benchmark programs. These memory traces would then be used to instrument a cache model and thereby establish the interference caused by OS and competing processes/threads on parallel applications. The measurements of interest to us would be: 
  • Time delay caused due to jitter : We need to simulate the Linux kernel, along with multi-threaded libraries, and precisely calculate the time lost to jitter.
  • Interference caused due to contention to shared resource : Multiple simultaneous threads will compete for shared resources like the memory-bus and cache. We wish to use a full system simulator along with detailed hardware models to characterize these effects.
  • Power overhead : Scaling up the frequency of a processor to compensate for jitter would increase power consumption of the cpu. We would need to model the resultant changes in performance very accurately.
Currently, we have constructed the prototype of the cache model we would like to use. Heuristics for converting jitter events to delay would requiring profiling on standard parallel banchmarks like Splash2.
